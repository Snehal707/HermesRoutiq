# HermesRoutiq ‚Äî Implementation Plan

Eleven phases. Each phase has an exit gate; you do not start the next phase until the gate passes. Lint + typecheck + unit tests run at every milestone; integration tests where applicable. Mocks are fine during development. The **final demo** must use the real sponsor surfaces (Hermes, Nemotron 3 Ultra, NemoClaw, cuOpt, Stripe Checkout/webhooks/Connect/Projects) ‚Äî no silent mock substitution.

## Phase 5 ‚Äî Routing service ‚úÖ DONE
FastAPI service at `services/routing/`; `RoutingProvider` interface; `MockRoutingProvider`/`OsrmRoutingProvider` (real road geometry via OSRM, no key required) + `CuOptRoutingProvider` (real NVIDIA cuOpt optimization). Default flow is `cuopt-osrm`: cuOpt decides driver assignment + stop order, OSRM provides the real road-following geometry between those optimized stops. Both the optimization decision (`routingPlan.assignedOrderIds`, `routingPlan.orderedStops`) and the resulting geometry are persisted per vehicle (`vehicles.routing_provider`, `vehicles.routing_plan` ‚Äî migration `0002_vehicle_routing_plan.sql`). Cached between loads via a Redis marker (`seed:42:provider:cuopt-osrm:v1`), re-fetched fresh on Reset. In-flight replans return a graceful `202 planning` state (Redis-backed single-owner lock) instead of transient 500s during the wait.
**Gate: PASSED.** Visual road-snapping confirmed (16-point fake paths ‚Üí 200+ point real geometry). cuOpt round-tripped against the real NVIDIA endpoint (`status=0`, 32 assignments, 0 unassigned). Before/after comparison proved cuOpt materially changes the displayed plan, not just geometry: stops redistributed across vehicles (e.g. vehicle-1 gained stops from vehicles 3/7/5); example route improved from 169 pts/538.7s/3494.2m to 52 pts/120.5s/783.7m. Reset confirmed to trigger fresh cuOpt+OSRM calls, not stale cache. Planning-state UX verified via Playwright: clean `202` handling, no console errors, no failed responses during a full Reset‚Üíplanning‚Üísettled cycle.
**Known remaining risk carried into later phases:** the graceful planning-state mechanism is verified for Reset only; it should also cover the future breakdown-recovery replan once Phase 6-8 wire Hermes/cuOpt into that flow, but that path doesn't exist yet so it's unverified. cuOpt replans are correctness-proven but not yet latency-optimized.

**Gate: PASSED.** 11/11 automated tests pass (tool discovery, malformed-input rejection on 3 tools, `$20` cap enforcement both directions, idempotency proven for `create_driver_payout` and `issue_customer_refund`, audit rows confirmed, skill-file write path exercised). Fixed by adding the `ws` package and wiring it into the shared client ‚Äî one fix at the client level covers all 16 tools' DB access path, verified by re-running the probe under a real Node 20 runtime (not just the contributor's local Node 22).

**Resulting architecture (honestly labeled, not overclaimed):**
1. **NemoClaw (real, sponsor-provided)** ‚Äî sandbox network/filesystem isolation, credential scoping (OpenRouter key never exposed inside the sandbox filesystem ‚Äî confirmed). ## Working rules
- Never generate the whole app in one step. Small testable milestones.
- Before each milestone: state which files will be created/changed.
- After each milestone: lint, typecheck, unit tests (integration where applicable).
- Snehal reviews prompts ‚Üí sends to Cursor (which has filesystem + built the codebase). Always read existing files/architecture before writing fix prompts. Never approve recursive deletes near `.git` or config. Stop for approval at each step of sensitive operations.
- Custom 3D vehicle models come last, after the deterministic business sim is solid.
